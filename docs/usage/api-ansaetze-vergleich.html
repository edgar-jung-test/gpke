Frankenstein API — Kommunikationsfluss / Communication flow
Vorteile / Advantages
Keine Prozessanpassung nötig — EDIFACT-Logik bleibt unverändert
Bestehende UTILMD-Dokumentation gilt weiter
Geringer Migrationsaufwand für Fachseite
Technisch einfach umsetzbar (nur Transportweg wechseln)
Nachteile / Disadvantages
Kein Request-Response — Antwort kommt als neue separate Anfrage
HTTP degeneriert zum reinen Transportmedium (wie AS4/E-Mail)
Keine sofortige fachliche Fehlerrückmeldung (Fehler im Body)
Kein Zustandsmechanismus (kein ETag, kein Status-Polling)
Keine Idempotenz — doppelter Versand erzeugt Duplikate
EDIFACT-Altlasten: hohe Einstiegshürde für neue Implementierungen
Kein modernes API-Tooling verwendbar
Technische Schulden — langfristig schwer wartbar
Nicht empfohlen API als Transportrohr. Alle Vorteile moderner APIs gehen verloren. HTTP-Standard wird missbraucht. Technische Schulden vorprogrammiert. Passend benannt: aus verschrotteten Altteilen erzwungen.

Fazit / Summary
REST API — Empfohlen

Der REST-Ansatz — so wie er in diesem Repository umgesetzt ist — ist der einzige der alle drei API-Versprechen gleichzeitig einlöst: einheitliche Schnittstelle, sofortiges Feedback, und Selbstbeschreibung.

Der Prüfidentifikator-Mechanismus von EDI@Energy lässt sich sauber auf operationId und HTTP-Statuscodes mappen, ohne EDIFACT-Semantik im Transport mitzuschleppen.

Das gesamte moderne API-Ökosystem steht zur Verfügung: OpenAPI-Generatoren, interaktive Dokumentation, Linting mit Spectral, Contract Testing — alles ohne zusätzlichen Aufwand.

Zustandsverwaltung über ETag + If-Match ist ein etabliertes, sicheres Muster für optimistic locking. Mehrschichtige Sicherheit (JWT + JWS + JWE + RFC 9421) ist vollständig integriert.

RPC API — Situationsabhängig

Der RPC-Ansatz ist ehrlicher als der Frankenstein-Ansatz, weil er offen zugibt dass es um Aktionen geht — nicht um Ressourcen. Er ist prozessnah und für interne Microservices (gRPC) hervorragend geeignet.

Für öffentliche EDI-APIs im deutschen Strommarkt ist er jedoch unüblich. Das Tooling-Ökosystem ist anders als bei REST, und bestehende EDI@Energy-Spezifikationen orientieren sich nicht an RPC-Semantik.

Als Brücke zwischen internem System und externem REST-Endpunkt kann RPC sinnvoll sein — aber nicht als das nach außen sichtbare API-Design.

Frankenstein API — Nicht empfohlen

Die Frankenstein API ist technisch gesehen kein API-Design, sondern ein Protokollwechsel auf Transportebene. Man tauscht AS4 oder E-Mail gegen HTTPS — die Nachrichtenstruktur und Prozesslogik bleiben unverändert.

Die eigentliche Erkenntnis: wenn man nur den Transportweg wechselt, kauft man sich alle Nachteile beider Welten — die Starrheit von EDIFACT und die Komplexität von HTTP — ohne die Vorteile einer der beiden zu nutzen.

Das Fehlen von Request-Response bedeutet: kein sofortiges Feedback, keine Validierung im Transport, keine Idempotenz, kein Zustandsmechanismus. Technische Schulden sind vorprogrammiert.

Der Name trifft es: aus verschrotteten Altteilen wird ein neues Konstrukt erzwungen — das zwar funktioniert, aber keiner modernen API-Architektur entspricht.

Empfehlung für EDI@Energy / Recommendation for EDI@Energy
Für den deutschen Strommarkt und die GPKE/UTILMD-Prozesse ist REST die klare Empfehlung. Die in diesem Repository umgesetzte Spezifikation (api/malo.yaml) zeigt, dass sich alle 7 Prozessschritte des SD: Lieferbeginn sauber auf REST-Ressourcen abbilden lassen — mit vollständiger Prüfidentifikator-Nachverfolgbarkeit, sofortiger Fehlerrückmeldung und dem gesamten modernen API-Ökosystem.
