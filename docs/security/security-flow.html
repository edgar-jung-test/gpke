OAuth 2.0 Bearer Token (JWT) RFC 7519
Authentifizierung / Authentication — jede Anfrage
▶

DE: Jede Anfrage muss ein gültiges OAuth 2.0 Bearer-Token im Authorization-Header tragen. Das Token ist ein JWT, das vom EDI@Energy-Autorisierungsserver ausgestellt wird und die Marktpartner-ID sowie Marktrollen enthält.

EN: Every request must carry a valid OAuth 2.0 Bearer token in the Authorization header. The token is a JWT issued by the EDI@Energy authorization server, containing the market partner ID and market roles.

JWS – JSON Web Signature RFC 7515
Payload-Integrität / Payload integrity — POST/PATCH Body
▶

DE: Der JSON-Payload wird mit ECDSA P-256 (ES256) signiert. Das JWS-Token im Compact Serialization Format besteht aus drei Base64url-Teilen: Header, Payload, Signatur.

EN: The JSON payload is signed with ECDSA P-256 (ES256). The JWS token in Compact Serialization format consists of three Base64url parts: header, payload, signature.

Algorithmus / Algorithm: ES256 (ECDSA with P-256 and SHA-256)

JWE – JSON Web Encryption RFC 7516
Payload-Vertraulichkeit / Payload confidentiality — optional
▶

DE: Das JWS-Token wird mit dem öffentlichen Schlüssel des Empfängers verschlüsselt. Algorithmus: RSA-OAEP-256 für den Schlüsselaustausch, A256GCM für die Inhaltsverschlüsselung. Content-Type wird zu application/jose.

EN: The JWS token is encrypted with the recipient's public key. Algorithm: RSA-OAEP-256 for key wrapping, A256GCM for content encryption. Content-Type becomes application/jose.

JWE Header: {"alg":"RSA-OAEP-256","enc":"A256GCM","cty":"JWT","kid":"hub-payload-enc-07"}

JWE Token (5 parts separated by .):
eyJhbGciOiJSU0EtT0FFUC0yNTYi...  ←  JWE Header
.m3kZ2...Q                         ←  Encrypted Key (JWS signing key encrypted with server pub key)
.48V1_ALb6US04U3b                  ←  Initialization Vector
.u9f0c...uWQ                       ←  Ciphertext (the JWS token, encrypted)
.o2mP...8w                         ←  Authentication Tag

HTTP Message Signature RFC 9421
Nachrichtenintegrität / Message integrity — alle Anfragen
▶

DE: Die gesamte HTTP-Nachricht wird signiert — Header-Felder plus den SHA-256-Hash des Body (Content-Digest). Schützt vor Replay-Attacken und Header-Manipulation. Zwei neue Header: Signature-Input (deklariert was signiert wird) und Signature (der Signaturwert).

EN: The entire HTTP message is signed — header fields plus the SHA-256 hash of the body (Content-Digest). Protects against replay attacks and header tampering. Two new headers: Signature-Input (declares what is signed) and Signature (the signature value).

Content-Digest: sha-256=:uU0p7h6qv6bqGJgYy3m8n2mY0cKc6lQ5qJv8W2r9bLk=:

Signature-Input: sig1=("@method" "@authority" "@target-uri"
  "content-type" "accept" "content-digest" "date"
  "x-request-id" "x-api-spec-ref" "x-api-spec-version");
  created=1769940930;expires=1769941230;
  keyid="client-http-sig-01";alg="ecdsa-p256-sha256"

Signature: sig1=:m1dP0w7qZ8QmM8n6f0dVbQ4h2lJd0q3H6mQkq9Vxq8g=:
GET ohne Body / GET without body — Header Signature only

DE: GET-Anfragen haben keinen Body, daher entfällt JWS/JWE. Nur der Bearer-Token und die HTTP-Message-Signatur sind erforderlich. Die Signatur deckt Methode, Pfad, Query-String und die wichtigsten Header ab.
EN: GET requests have no body, so JWS/JWE are not used. Only the Bearer token and the HTTP Message Signature are required. The signature covers method, path, query string and key headers.

Signatur-Konstruktion / Signature Construction
1
Kanonische Darstellung der signierten Komponenten erstellen (RFC 9421 §2.3)
Build canonical representation of signed components (RFC 9421 §2.3)
2
Signature Base String verketten: "@method": GET\n"@path": /malo/...\n...
3
Ed25519 oder ECDSA P-256 Signatur über den Base String berechnen
4
Base64url-kodieren und als sig1=:<value>: in den Signature-Header schreiben
POST mit JWS-signiertem Body / POST with JWS-signed body

DE: Der JSON-Payload wird als JWS signiert, bleibt aber lesbar (Content-Type: application/json). Zusätzlich wird der Body-Hash im Content-Digest-Header übertragen und die HTTP-Nachricht mit RFC 9421 signiert.
EN: The JSON payload is signed as JWS but remains readable (Content-Type: application/json). Additionally, the body hash is sent in the Content-Digest header and the HTTP message is signed with RFC 9421.

DE: Höchste Sicherheitsstufe. Der Payload wird zuerst als JWS signiert, dann als JWE verschlüsselt. Content-Type: application/jose. Der Content-Digest-Hash bezieht sich auf den verschlüsselten JWE-Token.
EN: Highest security level. The payload is first signed as JWS, then encrypted as JWE. Content-Type: application/jose. The Content-Digest hash covers the encrypted JWE token.
